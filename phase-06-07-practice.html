WhatsApp
Messaging
Hard

    ┌────────────▼─────────────┐

    └────────────┬─────────────┘

    │   (offline buffer)       │
    └────────────┬─────────────┘
                 │
    ┌────────────▼─────────────┐
    │   Delivery Workers       │  ← fan-out, push to online users
    └────────────┬─────────────┘
                 │
    ┌────────────▼─────────────┐
    │   Cassandra (messages)   │  ← partitioned by (user_id, timestamp)
    │   Redis (presence/cache) │
    │   S3 (media files)       │
    └──────────────────────────┘
Erlang/OTP Cassandra Kafka Redis AWS S3 WebSockets/XMPP
Why Cassandra for messages?
Messages are time-series data, extremely high write throughput (100B/day). Cassandra's wide-column model, partitioned by (user_id, time), is perfect. AP — we prefer availability over strict consistency for delivery receipts.
Why not store messages on server?
Messages are only stored temporarily while recipient is offline (store-and-forward). Once delivered, deleted from server. This enables E2E encryption — server only holds ciphertext transiently.
How do group messages scale?
Fan-out on write for small groups. For large groups (broadcast channels), use fan-out on read. Delivery workers fetch membership list and push to each online member's edge server.
How are delivery receipts done?
Each tick (1✓ server receipt, 2✓ device receipt, 2✓✓ read) is an ACK event published back through the system via Kafka, eventually delivered to the original sender's connection.

AWS S3 CDN (Akamai) Kafka Elasticsearch Redis HLS/DASH PostgreSQL
Why transcode to multiple resolutions?
Adaptive Bitrate Streaming (ABR). The player segments video into small chunks (2-10s each). It monitors available bandwidth and switches quality per segment. Poor connection → 360p, great connection → 4K. Seamless experience across all networks.
How is search so fast?
Elasticsearch maintains an inverted index of video titles, descriptions, and tags. When a video is uploaded, metadata is async-indexed via Kafka consumer. Search queries go directly to Elasticsearch, never to the main DB.
How are view counts handled?
View counts are extremely write-heavy. Instead of a DB write per view, events are batched in Kafka, aggregated in a stream processor (Flink/Spark), and periodically flushed to the database. Eventual consistency is fine — 10M vs 10,000,001 views looks the same to users.

The celebrity problem solution?
Hybrid fan-out. Regular users (<~100k followers): fan-out on write — tweet IDs pushed to all followers' Redis feed caches immediately. Celebrities: fan-out on read — tweets fetched on demand and merged with pre-built feed. Threshold is configurable.
How are trends calculated?
Redis Sorted Set where score = hashtag frequency. Each tweet increments the hashtag score. A background job maintains a 15-minute sliding window using ZINCRBY and ZREMRANGEBYSCORE. Top N from ZREVRANGE = trending.
Uber
Ride Matching
Hard
Functional
  • Rider: request ride, see ETA, track driver
  • Driver: go online/offline, accept/reject trips
  • Match nearest available driver to rider
  • Real-time location tracking
  • Fare calculation (surge pricing)
  • Trip history and payments
Non-Functional
  • Match in <1 minute or rider cancels
  • Location updates every 4 seconds
  • Highly consistent for payments
  • Millions of drivers, riders globally
  • Geospatial queries in milliseconds
Architecture
  DRIVER LOCATION UPDATES (every 4 seconds):
  Driver App → Location Service → Redis GEO  ← geospatial index
                                          → Kafka   ← async to persistent store
                                          → Cassandra (trip history)

  RIDE MATCHING:
  Rider requests → Matching Service
                        │
                   Query Redis GEO: GEORADIUS(rider_lat, rider_lon, 5km)
                        │  returns: [driver1, driver2, driver3...]
                        │
                   Score by: distance + rating + acceptance_rate
                        │
                   Send trip offer to best driver (WebSocket)
                        │
                   driver accepts → create Trip record → notify rider
                   driver rejects → offer to next driver (timeout: 10s)

  SURGE PRICING:
  Count active requests vs available drivers in geohash cell
  ratio > threshold → multiply base fare (1.2x, 1.5x, 2x...)

  PAYMENTS:  Stripe + PostgreSQL (ACID critical)
Redis GEO Kafka Cassandra PostgreSQL WebSockets Google Maps API
Why Redis GEO for location?
Redis GEO uses geohashing internally. GEORADIUS query returns all drivers within N km in O(N+log M) time, sub-millisecond for thousands of drivers. Driver locations update every 4s — Redis handles this write volume easily. Persistent store (Cassandra) updated async via Kafka.
What is geohashing?
Encodes a lat/lon pair into a short alphanumeric string. Nearby locations share the same prefix. Enables indexing geographic data in a regular key-value store or B-tree. Longer geohash = smaller area = more precision.

What makes Netflix CDN special?
Netflix built their own CDN — Open Connect Appliances (OCAs) — and placed them inside ISP data centres globally for free. ISPs get free hardware, Netflix gets ultra-low latency delivery without paying third-party CDN fees at their scale. OCAs are pre-loaded with popular content.
How do recommendations work at scale?
ML models run in batch (offline) every few hours, consuming viewing history, ratings, and content metadata from data warehouse. Recommendations are precomputed and stored per-user in Redis. Serving recommendations = Redis GET (1ms), not a model inference call.

How do Stories auto-expire?
Cassandra natively supports TTL (Time To Live) per row. When a story is written: INSERT INTO stories ... USING TTL 86400. After 86400 seconds (24h), Cassandra automatically marks and eventually deletes the row. No cron jobs needed — the DB handles it.
Why two databases (Cassandra + PostgreSQL)?
Use the right tool for the job. Cassandra: posts/stories — high write throughput, time-series, no complex joins needed. PostgreSQL: users/follows/likes — relational, needs joins ("show me all people user X follows who liked this post"), ACID for counts.
URL Shortener (Pastebin/TinyURL)
Utility
Easy
Functional
  • Given long URL → return short URL
  • Given short URL → redirect to long URL
  • Custom aliases (optional)
  • URL expiry (optional)
  • Click analytics
Non-Functional
  • 100M URLs created/day
  • Read-heavy: 100:1 read/write ratio
  • Redirect in <10ms
  • Short URLs: 7 chars
  • URLs must be unique, not predictable
Architecture
  ENCODE (long → short):
  1. Generate unique 7-char code:
     Option A: hash(long_url) → take first 7 chars of MD5/SHA256 (base62)
     Option B: auto-increment ID → convert to base62
     Base62 = [0-9 a-z A-Z] = 62^7 = 3.5 trillion unique URLs

  2. Store: short_code → { long_url, created_at, expires_at, user_id }
  3. Return: https://bit.ly/{short_code}

  DECODE (short → redirect):
  Request → Load Balancer
               │
          Check Redis Cache: GET short_code
          HIT:  Return 301/302 redirect immediately (~1ms)
          MISS: Query PostgreSQL → populate cache → redirect

  SCALE:
  Write: Single PostgreSQL (write volume is manageable)
  Read:  Redis cache (99% of redirects served from cache)
  Analytics: Async via Kafka → ClickHouse (analytics DB)
Redis PostgreSQL Kafka ClickHouse Base62 Encoding
301 vs 302 redirect?
301 (permanent): browser caches and never hits your server again — saves load but you lose click analytics. 302 (temporary): browser always hits your server — you capture analytics but more load. Most URL shorteners use 302 for analytics.
Hash collision handling?
Two URLs could hash to the same 7 chars. On write, check if the short code already maps to a different URL. If so, append a counter suffix and re-hash, or use the auto-increment approach (guaranteed unique) instead of content-hashing.
Google Drive
File Storage
Medium
Functional
  • Upload/download files of any type
  • Folder hierarchy
  • Share files/folders with permissions
  • Sync across devices
  • File versioning
  • Collaborative editing (Docs)
Non-Functional
  • 15GB free, petabytes total
  • Upload large files (>5GB)
  • Strong consistency (file write must be immediately visible)
  • High availability
  • Eventual sync across devices is acceptable
Architecture
  UPLOAD (chunked):
  Client splits file into 4MB chunks
  Each chunk: hash(chunk) → check if already exists (deduplication!)
  Upload missing chunks to Block Storage (GCS/S3)
  Metadata DB records: file_id, [chunk_hash_1, chunk_hash_2, ...]

  DOWNLOAD:
  Fetch chunk list from Metadata Service
  Download chunks (in parallel) from Block Storage
  Reassemble on client

  SYNC:
  File changed on Device A → upload delta → notify via WebSocket/SSE
  Device B receives notification → downloads changed chunks only

  DATA MODEL:
  Files:    MySQL { file_id, name, owner, parent_folder, version }
  Chunks:   MySQL { file_id, chunk_order, chunk_hash, size }
  Blocks:   GCS   { chunk_hash → binary data }  ← content-addressed
  Sharing:  MySQL { file_id, user_id, permission }

  VERSIONING:  Keep last N chunk lists per file, not full copies
GCS / S3 MySQL Redis Kafka WebSockets/SSE Content Addressing
Why chunk files?
Three reasons: (1) Resumable uploads — if connection drops, resume from last chunk, not beginning. (2) Delta sync — only upload changed chunks when a file is modified. (3) Deduplication — if chunk_hash already exists in block storage, don't re-upload it (saves storage across all users).
How is deduplication done?
Content-addressable storage: blocks are stored by their hash (SHA-256). Before uploading, client sends chunk hashes. Server checks which hashes already exist. Only missing chunks are uploaded. If two users upload the same file, the blocks are stored once and both metadata records point to the same blocks.

Video Conferencing
Hard
Functional

  • Screen sharing
  • In-meeting chat
  • Recording (local and cloud)
  • Virtual backgrounds
  • Breakout rooms
Non-Functional

  • High availability — call drops = bad UX

  • Handle poor network gracefully

Architecture
  1:1 CALL:
  Participant A ←──── WebRTC P2P ────► Participant B
  Direct peer-to-peer, Zoom server only does signalling
  Signalling server: exchange SDP (session description) + ICE candidates
  STUN: helps clients discover their public IP/port
  TURN: relay server when P2P is blocked by NAT/firewall

  GROUP CALL (3+ participants):
  Participant A ──► SFU (Selective Forwarding Unit) ◄── Participant B
                         │                              │
                         ▼                              ▼
                   Participant C                  Participant D
  SFU receives one stream from each participant.
  Forwards only the streams you need to see (active speaker + others at low res)
  Each participant uploads ONCE, receives N-1 streams

  SIGNALLING SERVICE:   WebSocket server (manages call state)
  MEDIA SERVERS (SFU):   Deployed in every region (low latency routing)
  RECORDING:            SFU records stream → S3 → async transcoding → CDN
WebRTC SFU (mediasoup/Janus) STUN/TURN WebSockets UDP AWS S3
SFU vs MCU vs P2P?
P2P: each participant sends video to every other (N×(N-1) streams). Breaks at 4+ people. MCU (Multipoint Control Unit): server decodes, mixes, re-encodes into one stream per participant — CPU-intensive. SFU (Selective Forwarding Unit): just forwards streams without decoding — much more scalable. Zoom/Meet use SFU for groups.
Why UDP over TCP for media?
A dropped video frame is acceptable — the next frame arrives in 33ms anyway. A dropped frame in TCP causes retransmit + head-of-line blocking, freezing all subsequent frames. In video calls, a brief visual artifact beats a frozen stream every time. WebRTC runs over SRTP (Secure RTP) over UDP.
Amazon (E-Commerce)
E-Commerce Platform
Hard
Functional
  • Product search and browse
  • Product detail pages
  • Shopping cart
  • Place order + checkout
  • Inventory management
  • Order tracking
Non-Functional
  • Millions of concurrent users
  • Inventory must be consistent (no overselling)
  • Cart can be eventually consistent
  • Search must be fast (<200ms)
  • Flash sales: 100x traffic spikes
Architecture
  PRODUCT SEARCH:
  Elasticsearch with product catalog indexed by name, category, brand
  Product updates sync via Kafka to keep index fresh

  CART SERVICE:
  Redis (session-based) ← fast, eventual consistency OK
  DynamoDB (persistent) ← survives session expiry, logged-out users

  ORDER PLACEMENT (most critical):
  1. Check inventory: SELECT FOR UPDATE on product (pessimistic lock)
   - or -  Optimistic locking: CAS on inventory count
  2. Reserve inventory → Create order record
  3. Process payment (Stripe) → Confirm order
  4. Publish "order_placed" event to Kafka
  All steps in one ACID transaction — SQL database

  INVENTORY SERVICE:
  PostgreSQL with row-level locks for inventory counts
  Redis cache for product page display (can be slightly stale)
  NEVER cache inventory for order placement — must be fresh!

  FLASH SALE HANDLING:
  Pre-warm caches, scale horizontally in advance
  Queue requests via Redis SETNX (atomic: first to set wins)
Elasticsearch PostgreSQL Kafka Redis DynamoDB Stripe
Optimistic vs Pessimistic locking for inventory?
Pessimistic (SELECT FOR UPDATE): lock the row while checking+decrementing — safe but serialises concurrent orders for the same product. Optimistic (CAS): read inventory, attempt update with WHERE inventory = old_value — retry on failure. Better for high-concurrency flash sales where contention is high.
How to handle flash sale traffic spikes?
Pre-scale horizontally. Use a queue (Redis or SQS) to absorb burst — accept order requests into queue, process at controlled rate. Pre-warm all caches. Use rate limiting to prevent any single user from hammering the system. Auto-scaling triggers based on CPU/request rate metrics.
