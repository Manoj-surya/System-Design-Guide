Topic 01
SQL Databases & ACID Properties

SQL (Structured Query Language) databases store data in structured tables with a fixed schema. Relationships between tables are defined via foreign keys. SQL databases are the backbone of most business-critical systems — payments, banking, e-commerce orders.

ACID Properties — The 4 Guarantees

ACID is what makes SQL databases trustworthy for critical data. Every transaction is guaranteed to follow all four properties.

⚛️
A — Atomicity

A transaction is all-or-nothing. If you transfer ₹1000 from A to B, either both the debit AND credit happen, or neither does. No partial states.

🔒
C — Consistency

The database moves from one valid state to another valid state. All rules, constraints, and relationships are enforced after every transaction.

🧱
I — Isolation

Concurrent transactions don't interfere with each other. Each transaction sees data as if it's the only one running (depending on isolation level).

💾
D — Durability

Once committed, a transaction is permanently saved — even if the server crashes immediately after. Data is written to disk (WAL logs).

Why Atomicity Matters — Bank Transfer Example
Transaction: Transfer ₹1000 from Alice → Bob

  Step 1: Debit  Alice  by ₹1000   ← success
  Step 2: Credit Bob    by ₹1000   ← server CRASHES here

Without Atomicity:  Alice lost ₹1000. Bob got nothing. Money vanished.
With Atomicity:    Both steps roll back. Alice's balance restored. Safe.
Indexes — Making Queries Fast

Without an index, a query like SELECT * FROM users WHERE email = 'alice@gmail.com' scans every row in the table — a full table scan. With a million rows, that's slow.

Index Concept
Without index:  Scan all 1,000,000 rows  →  O(n)  →  SLOW
With index:     B-Tree lookup on email   →  O(log n)  →  FAST

An index is a separate data structure (B-Tree or Hash) that stores
a sorted copy of a column's values + pointer to the full row.

Tradeoff:  Indexes speed up READs but slow down WRITEs
           (every insert/update must also update the index)
Rule:     Index columns you frequently filter/sort/join on

Topic 02
NoSQL — 4 Types Explained

NoSQL databases sacrifice some of SQL's guarantees (like strict schema and ACID) in exchange for flexibility, horizontal scalability, and speed at massive scale. There is no single "NoSQL" — there are four distinct types, each built for a different problem.

Type 01
Document DB
  • Stores JSON-like documents
  • Flexible, schema-less
  • Great for nested/variable data
  • MongoDB, CouchDB, Firestore
  • Use: User profiles, product catalogs
Type 02
Key-Value DB
  • Simplest model: key → value
  • Blazing fast reads/writes
  • Usually in-memory
  • Redis, DynamoDB, Memcached
  • Use: Sessions, caching, leaderboards
Type 03
Wide-Column DB
  • Rows + dynamic columns
  • Optimised for time-series data
  • Extremely high write throughput
  • Cassandra, HBase, Bigtable
  • Use: Logs, IoT, messaging history
Type 04
Graph DB
  • Nodes + edges + properties
  • Native relationship traversal
  • Cypher query language
  • Neo4j, Amazon Neptune
  • Use: Social graphs, fraud detection
SQL vs NoSQL — The Full Comparison
FactorSQLNoSQL
SchemaFixed, predefinedFlexible, dynamic
ACIDFull ACID guaranteesVaries — often eventual consistency
ScalingVertical (mostly)Horizontal (built for it)
JoinsNative, efficientNo joins — denormalise data instead
Query languageStandardised SQLVaries per DB (proprietary)
TransactionsMulti-row, multi-tableUsually single-document only
Best forPayments, orders, anything relationalScale, speed, unstructured data
The Key Mental Model

SQL normalises data to avoid duplication (separate tables, linked by keys). NoSQL denormalises — it duplicates data so reads are fast without joins. You trade storage efficiency for read speed. At internet scale, fast reads matter more than saving disk space.

Topic 03
Sharding & Partitioning

When a single database server can no longer handle your data volume or query load, you need to split your data across multiple database servers. This is called sharding (also known as horizontal partitioning).

The Problem Sharding Solves
Single DB server:
  ├─ 500 million users
  ├─ 10 TB of data
  ├─ 100,000 queries/second
  └─ One server can't handle this → bottleneck

After sharding (4 shards):
  ├─ Shard 1: users A–F   → 125M users, 2.5TB
  ├─ Shard 2: users G–M   → 125M users, 2.5TB
  ├─ Shard 3: users N–S   → 125M users, 2.5TB
  └─ Shard 4: users T–Z   → 125M users, 2.5TB
     Each shard handles only 25,000 queries/sec  ✓ manageable
3 Sharding Strategies
Range-Based Sharding

Split data by a range of values. Users A–F → Shard 1, G–M → Shard 2, etc. Simple to implement but creates hotspots if one range gets most traffic.

Hash-Based Sharding

Apply a hash function to the shard key (e.g. user_id % 4). Distributes data evenly. Problem: adding a new shard requires rehashing everything.

Directory-Based Sharding

A lookup table maps keys to shards. Most flexible — you can move data between shards without rehashing. But the lookup table itself becomes a bottleneck.

Hash-Based Sharding — How it Works
Shard key: user_id
Formula:   shard_number = hash(user_id) % total_shards

user_id = 1042  →  hash(1042) % 4  =  Shard 2
user_id = 5891  →  hash(5891) % 4  =  Shard 0
user_id = 2347  →  hash(2347) % 4  =  Shard 3

Every query for user_id=1042 always goes to Shard 2
No lookup needed — compute it on the fly
The Hotspot Problem

If your shard key is poorly chosen, one shard gets all the traffic. Example: sharding Instagram posts by celebrity user_id — one shard handles Cristiano Ronaldo's 600M followers while others sit idle. Always choose a shard key that distributes queries evenly.

Consistent Hashing — The Solution to Re-sharding

With regular hash-based sharding, adding one new shard means remapping nearly all keys (because user_id % 5 gives different results than user_id % 4). Consistent hashing arranges shards on a virtual ring and only remaps the keys that fall between the old and new shard — typically just 1/n of total keys. This is used by Cassandra, DynamoDB, and CDNs.

Topic 04
Replication & Migration

Replication means keeping copies of the same data on multiple servers. This gives you fault tolerance (if one server dies, others serve data) and read scalability (read from any replica).

3 Replication Models
Leader-Follower (Most Common)
         Leader (Primary)
              │
     ┌────────┼────────┐
     ▼        ▼        ▼
Follower1  Follower2  Follower3

Writes:  Always go to Leader only
Reads:   Can go to any Follower  ← scales read traffic
Problem: Leader dies → need to elect a new leader (failover)
Multi-Leader Replication
  Leader A  ←──────────────►  Leader B
(Data Center 1)              (Data Center 2)

Benefit:  Writes accepted in both data centers
          Low latency for geographically distributed users
Problem:  Write conflicts — what if both leaders modify same row?
          Need a conflict resolution strategy (last-write-wins, CRDT)
Leaderless Replication (Dynamo-style)
  Client writes to ALL (or majority of) replicas simultaneously

  Write quorum (w):  Must succeed on w nodes  e.g. 3 of 5
  Read quorum  (r):  Must read from r nodes   e.g. 3 of 5
  Rule:  w + r > n  →  guaranteed to read latest write

  Used by: Cassandra, DynamoDB, Riak
Sync vs Async Replication Trade-off

Synchronous: Leader waits for follower to confirm before acknowledging the write. Zero data loss but slower writes (you wait for network round-trip).

Asynchronous: Leader acknowledges immediately, replication happens in background. Faster writes but risk of data loss if leader crashes before follower catches up (replication lag).

Zero-Downtime Database Migration

Migrating a live production database without dropping requests is one of the hardest engineering challenges. Here's the standard approach:

Expand-Contract Migration Pattern
Phase 1 — Expand:
  Add new column (nullable) alongside old one
  Deploy code that writes to BOTH old and new column
  Old readers still work. No downtime.

Phase 2 — Backfill:
  Background job copies data from old → new column for existing rows
  Run in batches to avoid locking the table

Phase 3 — Switch:
  Deploy code that reads from new column
  Monitor for errors

Phase 4 — Contract:
  Remove old column (now unused)
  Migration complete with zero downtime
Topic 05
CAP Theorem

The CAP Theorem states: in a distributed system, you can only guarantee 2 of these 3 properties simultaneously. When a network partition occurs (and it always eventually will), you must choose between Consistency and Availability.

C
Consistency
Every read returns the most recent write. All nodes see the same data at the same time.
A
Availability
Every request gets a response (even if it might not have the latest data). System stays online.
P
Partition Tolerance
System continues operating even if network packets are lost between nodes.
Why P is Always Required

Network partitions (nodes losing ability to communicate) are inevitable in any distributed system. A node could fail, a cable could be cut, a data center could go offline. You cannot build a real-world distributed system that doesn't tolerate partitions — so P is non-negotiable. The real choice is always: CP or AP.

CP vs AP — Real System Examples
CP Systems — Consistency over Availability
  ├─ HBase, Zookeeper, MongoDB (by default)
  ├─ During partition: refuse to serve stale data → returns error
  └─ Used when: Banking, payments, inventory (never show wrong stock)

AP Systems — Availability over Consistency
  ├─ Cassandra, CouchDB, DynamoDB
  ├─ During partition: serve potentially stale data → always responds
  └─ Used when: Social media feeds, notifications, shopping carts

Example: WhatsApp notifications → AP
  It's OK if you see "2 grey ticks" a second late.
  Better than the app showing an error.

Example: Paytm balance → CP
  You must see your exact current balance.
  A 500 error is better than showing wrong balance.
Topic 06
Consistency Models & Isolation Levels
Consistency Models — The Spectrum

Consistency is not binary — it's a spectrum from "always perfectly up-to-date" to "eventually catches up". Each level trades off freshness for performance.

WEAKEST → Fastest STRONGEST → Slowest
Eventual Causal Linearizable
ModelGuaranteeUse Case
Strong / Linearizable Every read sees the most recent write, globally. Behaves as if there's only one copy of data. Bank balances, stock prices, inventory counts
Causal Consistency Operations that are causally related are seen by all in the same order. Unrelated ops can appear out of order. Comment replies (reply always appears after original post)
Quorum Consistency Read/write must succeed on majority of nodes (w + r > n). Middle ground between strong and eventual. Cassandra with tunable consistency, DynamoDB
Eventual Consistency If no new writes occur, all replicas will eventually converge to the same value. No time guarantee. DNS propagation, social media likes count, notifications
Eventual Consistency — Real Example

When you post a tweet, your follower in another country might not see it for a few seconds because their regional server hasn't synced yet — but it will eventually. That's eventual consistency. Twitter can tolerate this. Paytm cannot — your debit must be instantly visible everywhere.

Isolation Levels — 4 Levels in SQL

Isolation levels control how transactions interact with each other when running concurrently. Higher isolation = safer but slower (more locking overhead).

Read Uncommitted
Can read data that hasn't been committed yet (dirty reads). Fastest but most dangerous. Rarely used.
Read Committed
Only read committed data. Default in PostgreSQL. Prevents dirty reads but allows non-repeatable reads.
Repeatable Read
Same row read twice gives same result within a transaction. Prevents dirty + non-repeatable reads. Default in MySQL.
Serializable
Transactions execute as if sequential. Fully safe, zero anomalies. Slowest — heavy locking. Use for financial ops.
Problems Each Level Prevents
                     Dirty   Non-Rep  Phantom
                     Read    Read     Read
Read Uncommitted  →  ✗       ✗        ✗
Read Committed    →  ✓       ✗        ✗
Repeatable Read   →  ✓       ✓        ✗
Serializable      →  ✓       ✓        ✓

Dirty Read:       Read a value that another transaction later rolls back
Non-Repeatable:   Read same row twice, get different values (updated between reads)
Phantom Read:     Query returns different rows on two reads (rows inserted between reads)
Topic 07
When to Choose What

This is the most important output of Phase 2 — being able to justify your database choice in a system design interview. Here's the decision framework:

Database Decision Framework
Is your data relational with complex joins?
  YES → Start with SQL (PostgreSQL / MySQL)

Do you need ACID transactions (payments, orders)?
  YES → SQL (with Serializable or Repeatable Read isolation)

Do you need massive write throughput (logs, events, IoT)?
  YES → Cassandra (wide-column, AP, eventual consistency)

Do you need ultra-fast key lookups (sessions, caching)?
  YES → Redis (key-value, in-memory)

Do you have flexible/nested/variable schema (user profiles)?
  YES → MongoDB (document DB)

Do you need to traverse relationships (social graphs, fraud)?
  YES → Neo4j (graph DB)

Do you need global distribution with multi-region writes?
  YES → DynamoDB or Cassandra
SystemDatabase ChoiceWhy
WhatsApp (messages)CassandraHigh write throughput, time-series, AP
Uber (trips/payments)PostgreSQLACID, relational, consistency critical
Instagram (posts)PostgreSQL + CassandraSQL for users/follows, Cassandra for feeds
Netflix (user prefs)Cassandra + RedisScale + caching of recommendations
LinkedIn (connections)Neo4j or custom graphRelationship traversal (2nd/3rd degree connections)
Any app sessionsRedisFast key-value, TTL expiry, in-memory
Interview Tip — Always Justify

Never just say "I'll use MongoDB". Say: "I'll use Cassandra because we need high write throughput for 10M messages per second, the data is time-series in nature (message_id + timestamp), and we can tolerate eventual consistency since a slight delay in delivery receipts doesn't affect UX." This is what separates good candidates from great ones.

Key Takeaway — Phase 02

There is no perfect database. Every choice is a trade-off.
Know what you're sacrificing when you choose one,
and justify it against your system's actual requirements.
