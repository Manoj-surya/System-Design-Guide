01
Four Pillars of OOP

Object-Oriented Programming organises code around objects that bundle data and behaviour. These four pillars are the foundation of every LLD interview — you'll be expected to apply them, not just define them.

E
Encapsulation

Hide internal state. Expose only what's needed via public methods. Prevents external code from directly modifying internal data.

A
Abstraction

Show only what something does, hide how it does it. You use a TV remote without knowing its circuits.

I
Inheritance

Child classes inherit behaviour from parent. Enables code reuse. Dog extends Animal — gets eat() and sleep() for free.

P
Polymorphism

Same interface, different behaviour. Call draw() on Circle, Square, Triangle — each behaves differently. One call, many forms.

All 4 Pillars in One Example
class BankAccount {
  private balance: number   // ENCAPSULATION: hidden, not directly accessible

  constructor(initial: number) { this.balance = initial }

  public deposit(amount: number): void {  // public interface only
    if (amount > 0) this.balance += amount  // validation hidden inside
  }

  public getBalance(): number { return this.balance }  // controlled read
}

class SavingsAccount extends BankAccount {  // INHERITANCE: reuses deposit/getBalance
  private interestRate = 0.05

  applyInterest() { this.deposit(this.getBalance() * this.interestRate) }
}

// ABSTRACTION: caller doesn't know HOW deposit works internally
// POLYMORPHISM: deposit() behaves differently in SavingsAccount if overridden
02
SOLID Principles

SOLID is an acronym for five design principles that make software more maintainable, extensible, and testable. They are the single most important set of guidelines for LLD interviews.

S
Single Responsibility Principle
A class should have only ONE reason to change. One class = one job. If a class handles user auth AND sends emails AND writes to DB, that's three responsibilities.
✗ class UserManager { login() + sendEmail() + saveToDb() }
✓ AuthService + EmailService + UserRepository — each one thing
O
Open/Closed Principle
Open for extension, closed for modification. Add new behaviour by adding new code, not changing existing code. Prevents breaking what already works.
✗ Adding new payment type requires editing PaymentProcessor's if-else chain
✓ PaymentProcessor uses interface PaymentMethod — add CryptoPayment without touching existing code
L
Liskov Substitution Principle
Subtypes must be substitutable for their base types without breaking the program. If S extends T, anywhere T is used, S must work correctly there too.
✗ class Penguin extends Bird { fly() { throw Error("can't fly") } } — breaks callers of Bird
✓ Separate FlyingBird and NonFlyingBird interfaces — Penguin only implements what it can do
I
Interface Segregation Principle
Clients shouldn't depend on interfaces they don't use. Many small, specific interfaces are better than one fat interface that forces implementing unused methods.
✗ interface Worker { work() + eat() + sleep() } — RobotWorker forced to implement eat()/sleep()
✓ interface Workable { work() } + interface Livable { eat(); sleep() }
D
Dependency Inversion Principle
High-level modules shouldn't depend on low-level modules. Both should depend on abstractions. Depend on interfaces, not concrete implementations.
✗ class OrderService { db = new MySQLDatabase() } — tightly coupled to MySQL
✓ class OrderService { db: IDatabase } — inject any DB (MySQL, Postgres, Mock) from outside
Why SOLID Matters in Interviews

When you write LLD code in an interview and the interviewer asks "how would you add a new payment method?" or "how would you make this testable?" — SOLID is your answer. Every SOLID principle directly enables either extensibility (O, D), testability (D, S), or correctness (L, I).

03
Design Patterns

Design patterns are proven, reusable solutions to commonly recurring problems in software design. There are 23 classic Gang of Four patterns split into three categories. Know all 18 listed below — their intent, when to use, and a concrete example.

CREATIONAL
5 PATTERNS — HOW OBJECTS ARE CREATED
Singleton
Ensures a class has only ONE instance and provides a global access point to it.
→ Database connection pool, Logger, Config manager, Thread pool
class DBPool { private static instance: DBPool private constructor() {} // blocks new DBPool() static getInstance() { if (!DBPool.instance) DBPool.instance = new DBPool() return DBPool.instance } }
Factory Method
Defines an interface for creating objects, but lets subclasses decide which class to instantiate. Delegates object creation to subclasses.
→ UI components (Button → WindowsButton / MacButton), parsers, loggers
abstract class Notification { abstract createSender(): ISender // factory method send(msg) { this.createSender().send(msg) } } class EmailNotification extends Notification { createSender() { return new EmailSender() } }
Abstract Factory
Creates families of related objects without specifying their concrete classes. A factory of factories — ensures objects from the same family are used together.
→ Cross-platform UI kits (Windows/Mac/Linux), database drivers, theme systems
interface GUIFactory { createButton(): Button createCheckbox(): Checkbox } class MacFactory implements GUIFactory { createButton() { return new MacButton() } createCheckbox() { return new MacCheckbox() } }
Builder
Constructs complex objects step-by-step. Separates construction from representation. Useful when an object needs many optional configuration parameters.
→ SQL query builder, HTTP request builder, complex config objects, meal ordering
const query = new QueryBuilder() .select("id", "name") .from("users") .where("age > 18") .orderBy("name") .limit(10) .build() // No telescoping constructor with 8 parameters
Prototype
Creates new objects by copying (cloning) an existing object. Avoids costly re-initialisation when a similar object already exists.
→ Game objects (clone enemy template), document templates, object caching
class Monster { constructor(public hp, public attack, public name) {} clone() { return new Monster(this.hp, this.attack, this.name) } } const goblinTemplate = new Monster(100, 15, "Goblin") const goblin1 = goblinTemplate.clone() // fast!
STRUCTURAL
6 PATTERNS — HOW CLASSES ARE COMPOSED
Adapter
Converts the interface of one class into another interface that clients expect. Makes incompatible interfaces work together — like a power socket adapter.
→ Integrating legacy code, third-party libraries, different API versions
// Old logger: log(message: string) // New system expects: write(level, msg) class LoggerAdapter implements INewLogger { constructor(private old: OldLogger) {} write(level, msg) { this.old.log(`[${level}] ${msg}`) } }
Decorator
Adds new behaviour to an object dynamically by wrapping it. Extends functionality without subclassing. Decorators can be stacked.
→ Java I/O streams, middleware pipelines, auth wrappers, logging wrappers
class LoggingCoffee implements Coffee { constructor(private base: Coffee) {} cost() { return this.base.cost() + 0.5 } description() { return this.base.description() + " + log" } } // Stack: new Milk(new Sugar(new LoggingCoffee(baseCoffee)))
Facade
Provides a simplified interface to a complex subsystem. Hides internal complexity behind one easy-to-use class.
→ Home theatre systems, compiler front-end, payment SDK, hotel booking
// Without Facade: caller must orchestrate 5 steps class HomeTheatreFacade { watchMovie(movie: string) { this.dvd.on(); this.projector.on() this.amplifier.on(); this.dvd.load(movie) this.projector.setInput("dvd"); this.dvd.play() } }
Proxy
Provides a surrogate/placeholder for another object to control access to it. Can add lazy loading, access control, caching, logging before delegating to real object.
→ Lazy loading of heavy objects, API rate limiting, caching proxy, auth guard
class ImageProxy implements Image { private realImage: RealImage | null = null display() { if (!this.realImage) { this.realImage = new RealImage(this.url) // lazy load } this.realImage.display() } }
Composite
Composes objects into tree structures to represent part-whole hierarchies. Treats individual objects and compositions of objects uniformly.
→ File system (files + folders), UI component trees, org charts, DOM
interface FileSystem { getSize(): number } class File implements FileSystem { getSize() { return this.size } } class Folder implements FileSystem { children: FileSystem[] = [] getSize() { return this.children.reduce((s, c) => s + c.getSize(), 0) } }
Bridge
Decouples an abstraction from its implementation so both can vary independently. Avoids a permanent binding between abstraction and implementation.
→ UI rendering engines, device drivers, cross-platform APIs
// Without Bridge: RemoteWindowsTV, RemoteLinuxTV, // SmartRemoteWindowsTV... exponential classes class Remote { constructor(private tv: TV) {} // Bridge: inject implementation togglePower() { this.tv.on() } } // Now Remote and TV vary independently
BEHAVIOURAL
7 PATTERNS — HOW OBJECTS COMMUNICATE
Observer
Defines a one-to-many dependency. When one object (subject) changes state, all its dependents (observers) are notified and updated automatically.
→ Event systems, UI frameworks (React state), notifications, stock price alerts
class EventEmitter { private listeners: Map<string, Function[]> = new Map() on(event, fn) { this.listeners.get(event)?.push(fn) } emit(event, data) { this.listeners.get(event)?.forEach(fn => fn(data)) } } // WhatsApp: MessageSubject notifies ReadReceiptObserver, NotificationObserver
Strategy
Defines a family of algorithms, encapsulates each one, and makes them interchangeable. Lets the algorithm vary independently from clients that use it.
→ Sorting algorithms, payment methods, compression algorithms, route planning
interface SortStrategy { sort(data: number[]): number[] } class QuickSort implements SortStrategy { sort(d) { /* ... */ } } class MergeSort implements SortStrategy { sort(d) { /* ... */ } } class Sorter { constructor(private strategy: SortStrategy) {} sort(data) { return this.strategy.sort(data) } } // Switch strategy at runtime based on data size
Command
Encapsulates a request as an object. Allows parameterising clients with different requests, queuing operations, logging, and supporting undo/redo.
→ Undo/redo, macro recording, task queues, transaction logs, GUI buttons
interface Command { execute(): void; undo(): void } class MoveCommand implements Command { constructor(private piece, private from, private to) {} execute() { this.piece.moveTo(this.to) } undo() { this.piece.moveTo(this.from) } } // Chess game: push commands to history stack, pop to undo
State
Allows an object to alter its behaviour when its internal state changes. The object will appear to change its class. Replaces large if-else / switch chains.
→ Vending machine, ATM states, order lifecycle (pending→paid→shipped), traffic lights
// Order state machine interface OrderState { processPayment(): void; ship(): void } class PendingState implements OrderState { processPayment() { order.setState(new PaidState()) } ship() { throw new Error("Pay first!") } } class PaidState implements OrderState { processPayment() { throw new Error("Already paid") } ship() { order.setState(new ShippedState()) } }
Template Method
Defines the skeleton of an algorithm in a base class but lets subclasses override specific steps without changing the algorithm's overall structure.
→ Data processing pipelines, game turn sequences, report generators, HTTP handlers
abstract class DataMigration { // Template method — fixed order run() { this.connect() const data = this.extract() const transformed = this.transform(data) this.load(transformed) this.disconnect() } abstract extract(): any[] // subclasses override abstract transform(d): any[] // subclasses override }
Iterator
Provides a way to sequentially access elements of a collection without exposing its underlying representation (array, tree, graph, etc.).
→ Collection traversal, database result sets, file system walking, pagination cursors
class NumberRange { constructor(private start, private end) {} [Symbol.iterator]() { let current = this.start return { next: () => current <= this.end ? { value: current++, done: false } : { done: true } } } } for (const n of new NumberRange(1, 5)) { console.log(n) }
Chain of Responsibility
Passes a request along a chain of handlers. Each handler decides either to process the request or pass it to the next. Decouples sender from receivers.
→ Middleware pipelines (Express.js), support ticket escalation, auth/rate-limit/logging chains
abstract class Handler { protected next: Handler | null = null setNext(h: Handler) { this.next = h; return h } handle(req: Request) { this.next?.handle(req) } } class AuthHandler extends Handler { handle(req) { if (!req.token) return respond(401) super.handle(req) // pass to next handler } }
Interview Strategy for Design Patterns

Don't memorise patterns in isolation. Learn to spot which problem calls for which pattern: need one instance? Singleton. Need to swap algorithms? Strategy. Need notifications? Observer. Need to add features without changing existing code? Decorator or Visitor. Interviewers test recognition and application, not recitation.

Concurrency & Thread Safety

When multiple threads access shared data simultaneously, things go wrong in subtle, hard-to-reproduce ways. Understanding concurrency is essential for designing correct multi-threaded systems.

Race Condition — The Classic Problem
// balance = 1000. Two threads simultaneously:
Thread A:  read balance (1000)
Thread B:  read balance (1000)
Thread A:  balance = 1000 - 200 = 800  → write 800
Thread B:  balance = 1000 - 300 = 700  → write 700  ← overwrites A!

Result: 700. Expected: 500. ₹200 created from thin air.

Fix: Lock the balance before reading, unlock after writing
Thread A:  LOCK → read(1000) → write(800) → UNLOCK
Thread B:  waits... LOCK → read(800) → write(500) → UNLOCK  ✓
Core Synchronisation Primitives
Mutex (Lock)

Mutual exclusion. Only one thread can hold the lock at a time. Others block until it's released. Use for protecting shared mutable state.

Semaphore

Like a mutex but with a count. Allows N threads simultaneously. A semaphore(3) lets 3 threads access a resource pool concurrently.

Monitor

Higher-level lock that bundles a mutex + condition variables. Java's synchronized keyword is a monitor. Enables wait/notify patterns.

Atomic Operations

CPU-level instructions that execute as one indivisible operation. AtomicInteger, CompareAndSwap. Lock-free concurrency for simple counters.

Deadlock

Thread A holds Lock 1, waits for Lock 2. Thread B holds Lock 2, waits for Lock 1. Neither can proceed — system frozen. Solution: always acquire locks in same order.

Thread Pool

Pre-create N threads and reuse them for tasks. Avoids cost of creating/destroying threads per task. ExecutorService in Java, Worker threads in Node.js.

Producer-Consumer Pattern
Producer-Consumer with Bounded Buffer
// Classic concurrent pattern: one thread produces, another consumes
// Shared: a bounded queue (max size N)

class BoundedQueue {
  private queue: Task[] = []
  private maxSize = 10

  produce(task: Task) {
    synchronized(this) {
      while (queue.length === maxSize) wait()  // queue full — sleep
      queue.push(task)
      notifyAll()  // wake consumers
    }
  }

  consume(): Task {
    synchronized(this) {
      while (queue.length === 0) wait()  // queue empty — sleep
      const task = queue.shift()!
      notifyAll()  // wake producers
      return task
    }
  }
}
// This is exactly how message queues (Kafka/RabbitMQ) work under the hood
05
API Design Principles

In LLD interviews you'll often be asked to design the APIs for a system. A well-designed API is intuitive, consistent, and extensible. These are the principles that matter.

REST Conventions

Nouns not verbs in URLs. GET /users not GET /getUsers. HTTP methods carry the verb: GET=read, POST=create, PUT=replace, PATCH=partial update, DELETE=remove.

Idempotency

GET, PUT, DELETE are idempotent — calling them N times = same result as once. POST is not. Critical for safe retries — PUT /orders/123 can be retried without creating duplicates.

Versioning

Never break existing clients. Version your API: /v1/users, /v2/users. Or use Accept-Version header. Keep v1 alive until all clients migrate. Breaking changes go in new version.

Pagination

Never return all records. Offset pagination: ?page=2&limit=20. Cursor pagination: ?cursor=xyz&limit=20. Cursor is more efficient for large datasets (no count query needed).

Error Handling

Use correct status codes: 400 (bad input), 401 (not authenticated), 403 (not authorised), 404 (not found), 409 (conflict), 429 (rate limited), 500 (server error). Include error detail in body.

Request/Response Models

Never expose internal DB models directly. Create dedicated DTOs (Data Transfer Objects) for input and output. Separate concerns between persistence and transport layers.

Good API Design — Example
// Bad API
POST /getUser?userId=42
POST /createNewUser
POST /deleteUser?id=42&confirm=true

// Good RESTful API
GET    /users/42          → return user 42
POST   /users             → create user (body has user data)
PATCH  /users/42          → partial update user 42
DELETE /users/42          → delete user 42
GET    /users?page=2&limit=20  → paginated list

// Response envelope
{
  "data": { "id": 42, "name": "Alice" },
  "meta": { "page": 2, "total": 150 },
  "error": null
}
06
Clean Code Principles

LLD interviews evaluate not just whether your code works, but whether it's readable, maintainable, and well-structured. These principles are the difference between code that works and code that teams can build on.

DRY

Don't Repeat Yourself. Every piece of knowledge should have one authoritative source. Duplication means two places to update and two places to get out of sync.

KISS

Keep It Simple. The simplest solution that works is the best. Don't pre-optimise, don't over-engineer. Simple code is easier to test, read, and modify.

YAGNI

You Aren't Gonna Need It. Don't build features "just in case". Build what's needed now. Every premature abstraction adds complexity that you'll maintain forever.

Avoid God Classes

A class that knows and does too much. UserManager that handles auth + profiles + billing + notifications. Break it down. If it's hard to name, it's doing too much.

Small Functions

Functions should do ONE thing and do it well. If you can't describe what a function does in one sentence without "and", split it. Aim for under 20 lines per function.

Meaningful Names

Names should reveal intent. calculateInvoiceTotal() beats calc(). isUserActive beats flag. Good names eliminate the need for most comments.

LLD Interview Checklist

1. Clarify requirements first — functional (what it does) and constraints (scale, edge cases)
2. Identify the core entities and their relationships
3. Define interfaces before writing classes (depend on abstractions — DIP)
4. Explicitly name which design patterns you're using and why
5. Show that your design handles adding new features without touching old code (OCP)
6. Think about thread safety if the system is concurrent
7. Be ready to write unit tests — testable code = good OOP code
