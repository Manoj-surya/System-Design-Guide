Topic 01
Caching — How & Why

A cache is a fast, temporary storage layer that sits between your application and your database. Its purpose is simple: instead of hitting the database on every request, store frequently-accessed data in memory so it can be served in microseconds instead of milliseconds.

Without Cache vs With Cache
Without Cache:
  User requests profile  →  App  →  Database  →  App  →  User
  Latency: ~100–200ms (disk read + query execution)
  At 10,000 req/sec: database gets hammered, becomes bottleneck

With Cache:
  User requests profile  →  App  →  Cache HIT  →  User
  Latency: ~1ms (memory read)
  Database only gets called on CACHE MISS (first request or expired data)
Redis vs Memcached
FeatureRedisMemcached
Data structuresStrings, Hashes, Lists, Sets, Sorted Sets, StreamsOnly strings (key-value)
PersistenceOptional (RDB snapshots, AOF logs)None — purely in-memory
ReplicationMaster-replica, Sentinel, ClusterNo built-in replication
Pub/SubYes — built-inNo
Lua scriptingYesNo
Multi-threadingSingle-threaded (mostly)Multi-threaded
Best forEverything — sessions, queues, leaderboards, rate limitingSimple, high-speed object caching only
Always Default to Redis

In 2024+, Redis is the industry default for caching. Memcached is faster at simple string lookups under very specific conditions, but Redis's rich data structures, persistence, and pub/sub make it the right choice for almost all system design scenarios. When in doubt, say Redis.

Where to Place a Cache
Cache Placement in a System
Client-Side Cache   →  Browser cache, mobile app cache
                       Fastest — no network needed
                       Limited control over invalidation

CDN Cache          →  Edge servers near users (Phase 3 - Topic 4)
                       Great for static assets (images, JS, video)

Application Cache  →  In-process memory (e.g. local HashMap in Node.js)
                       Fastest server-side option but not shared across servers
                       Breaks with horizontal scaling

Distributed Cache  →  Redis / Memcached  ← Most common in system design
                       Shared across ALL app servers
                       Works perfectly with horizontal scaling

Database Cache     →  Query result cache built into MySQL/PostgreSQL
                       Automatic but limited control
Cache Write Policies

When data is written, how do we keep the cache and database in sync? There are three strategies, each with different trade-offs between consistency and performance.

Write Policy 01
Write-Through

Write to cache AND database simultaneously on every write. Cache is always consistent with DB. Slower writes (both must succeed) but zero stale data risk.

Write Policy 02
Write-Back (Write-Behind)

Write to cache only, acknowledge immediately. Database is updated asynchronously in the background. Fastest writes but risk of data loss if cache dies before DB sync.

Write Policy 03
Write-Around

Write directly to database, bypass cache entirely. Cache is only populated on reads. Avoids polluting cache with write-heavy data rarely read again.

Read Strategy
Cache-Aside (Lazy Loading)

App checks cache first. On miss, reads from DB and populates cache. Most flexible strategy — app controls what gets cached. Industry default pattern.

Cache-Aside Pattern — Code Flow
function getUserProfile(userId):
  // Step 1: Check cache
  data = redis.get(`user:${userId}`)
  
  if data exists:
    return data  ← CACHE HIT (fast, ~1ms)
  
  // Step 2: Cache miss — go to database
  data = database.query(`SELECT * FROM users WHERE id = ${userId}`)
  
  // Step 3: Populate cache for next time
  redis.set(`user:${userId}`, data, TTL=3600)  ← expires in 1 hour
  
  return data  ← CACHE MISS (slow, ~100ms, but only first time)
Cache Eviction Algorithms

Cache has limited memory. When it's full and a new item needs to be stored, something must be evicted. The algorithm decides which item to remove.

Eviction 01
LRU

Least Recently Used. Evict the item that hasn't been accessed for the longest time. Most popular. Works well for general-purpose caching.

Eviction 02
LFU

Least Frequently Used. Evict the item accessed the fewest times overall. Better than LRU when access patterns are stable (popular items stay forever).

Eviction 03
Segmented LRU

Two-tier LRU. New items enter a probationary segment. Only items accessed again move to the protected segment. Protects hot items from one-time scan eviction.

Eviction 04
TTL (Time-Based)

Time To Live. Every item expires after a set time regardless of access. Simplest form of invalidation. Used alongside LRU/LFU in practice.

Cache Stampede — The Hidden Danger

Imagine 10,000 users all request a popular item at the exact moment its TTL expires. All 10,000 requests find a cache miss simultaneously and all hit the database at once — potentially crashing it. Solutions: Jitter (randomise TTL slightly so not all expire together), Probabilistic early expiration (start refreshing before TTL ends), or Lock (only one request refreshes the cache, others wait).

Topic 02
CDN — Content Delivery Networks

A CDN is a globally distributed network of servers (called Points of Presence / Edge Servers) that cache and serve static content from a location physically close to the user. Instead of every user hitting your origin server in Mumbai, a user in London gets content from a CDN edge server in London.

CDN Request Flow
Without CDN:
  User (London) → Request → Origin Server (Mumbai) → Response
  Latency: ~200ms  (physical distance + network hops)

With CDN:
  User (London) → Request → CDN Edge (London) → Response
  Latency: ~5ms   (content already cached at nearest edge)

  On first request (cache miss):
  CDN Edge (London) → pulls from Origin (Mumbai) → caches it → serves user
  All subsequent requests served from edge cache
Push CDN vs Pull CDN
Push CDN
  • You manually upload assets to CDN in advance
  • Content is there before anyone requests it
  • No cache miss on first request
  • Good for: large video files, known-popular content
  • Netflix pre-pushes new episodes before release
  • You control when content is refreshed
Pull CDN
  • CDN fetches from origin on first request (cache miss)
  • Caches and serves from edge for subsequent requests
  • Zero setup — just point CDN at your origin
  • Good for: general websites, unpredictable traffic
  • Cloudflare, AWS CloudFront default mode
  • First request is slower (origin fetch overhead)
What CDNs Cache

CDNs are ideal for static, immutable content: images, videos, CSS files, JavaScript bundles, fonts, HTML pages. They are NOT suitable for dynamic, user-specific data (like a user's inbox or personalised feed) because that can't be cached globally — every user needs different data.

Topic 03
Load Balancers

A load balancer distributes incoming requests across multiple servers to ensure no single server gets overwhelmed. It is the entry point for all traffic into your server cluster and is what makes horizontal scaling possible.

Load Balancer in a System
                       ┌─────────────────────┐
  Internet Traffic  →  │    Load Balancer     │  ← Single entry point
                       └──────┬──────┬────────┘
                              │      │      │
                          Server1  Server2  Server3
                         (healthy) (healthy) (healthy)

  If Server2 goes down:
                       └──────┬─────────────┘
                              │             │
                          Server1        Server3
                         (traffic auto-redistributed, no downtime)
L4 vs L7 Load Balancing
L4 — Transport Layer
  • Routes based on IP address + TCP/UDP port only
  • Doesn't inspect packet contents — very fast
  • Cannot route based on URL path or HTTP headers
  • Lower overhead — great for raw throughput
  • Example: AWS Network Load Balancer (NLB)
L7 — Application Layer
  • Inspects HTTP headers, URL, cookies, body content
  • Can route /api → API servers, /images → image servers
  • Supports SSL termination, A/B testing, auth
  • Slightly more overhead but far more powerful
  • Example: AWS ALB, Nginx, HAProxy
Load Balancing Algorithms
Round Robin

Requests distributed sequentially: 1→S1, 2→S2, 3→S3, 4→S1... Simple, equal distribution. Bad if servers have different capacities.

Weighted Round Robin

Servers get requests proportional to their weight. S1 (weight 3) gets 3x more than S2 (weight 1). Great for heterogeneous server fleet.

Least Connections

Next request goes to server with fewest active connections. Best for requests with variable processing time (some fast, some slow).

IP Hash

hash(client_IP) % servers → same client always hits same server. Creates sticky sessions. Useful when server-side state exists.

Consistent Hashing

Like IP hash but handles server addition/removal gracefully. Only ~1/n keys remapped when topology changes. Used in distributed caches.

Random

Pick a random server. Statistically converges to equal distribution at scale. Dead simple and surprisingly effective for large fleets.

Reverse Proxy vs Forward Proxy
Proxy Types
Forward Proxy  — sits in FRONT of clients
  Client → Forward Proxy → Internet
  Use: VPNs, corporate firewalls, anonymity
  The server doesn't know which client made the request

Reverse Proxy  — sits in FRONT of servers
  Client → Reverse Proxy → Server
  Use: Load balancing, SSL termination, caching, DDoS protection
  The client doesn't know which backend server responded
  Nginx, HAProxy, AWS ALB are all reverse proxies
Rate Limiting Algorithms

Rate limiting prevents any single client from overwhelming your system. It's usually implemented at the load balancer or API gateway level. There are four common algorithms:

Token Bucket

A bucket holds tokens. Each request consumes 1 token. Tokens are added at a fixed rate. If bucket is empty, request is rejected. Allows bursts up to bucket size.

Leaky Bucket

Requests go into a queue (bucket). Processed at a fixed rate — like water leaking at constant speed. Smooths out bursts. If queue full, requests dropped.

Fixed Window Counter

Count requests per fixed time window (e.g. 100 req/min). Simple but has edge case: 100 at 0:59 + 100 at 1:01 = 200 in 2 seconds. Window boundary exploit.

Sliding Window Log

Store timestamp of each request. On each request, count how many timestamps fall within the last window. Most accurate but memory-intensive (stores all timestamps).

Refill rate: 1 token/second

Topic 04
Message Queues — Kafka vs RabbitMQ

Message queues decouple producers (who send messages) from consumers (who process them). They enable asynchronous processing — the producer doesn't wait for the consumer to finish. This is critical for handling spikes, ensuring reliability, and separating concerns.

Pub/Sub Model — The Core Pattern
Publisher → publishes message to a Topic
             ↓
         [Message Queue]   ← stores messages durably on disk
             ↓
  ┌──────────┼──────────┐
  ▼          ▼          ▼
Consumer1  Consumer2  Consumer3  ← each reads independently

Example: User uploads a photo on Instagram
  Publisher → "photo_uploaded" event → Queue
  Consumer1 → resize to thumbnail
  Consumer2 → run content moderation AI
  Consumer3 → notify followers
  All three happen in parallel, user doesn't wait for any of them
Kafka vs RabbitMQ — Deep Comparison
Apache Kafka
  • Distributed commit log — messages stored durably on disk
  • Messages are NOT deleted after consumption (retained for days/weeks)
  • Consumers track their own offset — can replay old messages
  • Partitions allow massive parallelism (millions/sec throughput)
  • Consumer groups — multiple consumers share a topic's partitions
  • Best for: event streaming, audit logs, real-time analytics
  • Used by: LinkedIn, Uber, Airbnb, Netflix
RabbitMQ
  • Traditional message broker with exchanges and queues
  • Messages deleted once consumed and acknowledged
  • Flexible routing: direct, fanout, topic, headers exchanges
  • Built-in dead letter queues for failed messages
  • Lower throughput than Kafka but simpler setup
  • Best for: task queues, job processing, microservice communication
  • Used by: many mid-size applications
Kafka Architecture — Key Concepts
Topic: "user-events"
  ├─ Partition 0:  [msg1] [msg4] [msg7]  ← Consumer Group A reads this
  ├─ Partition 1:  [msg2] [msg5] [msg8]  ← Consumer Group A reads this
  └─ Partition 2:  [msg3] [msg6] [msg9]  ← Consumer Group A reads this

Each consumer in a group gets one or more partitions
More partitions = more parallelism = higher throughput
Offset: each consumer remembers where it left off (can replay from offset 0)

At-least-once delivery:  consumer crashes → message reprocessed from last committed offset
Exactly-once delivery:   Kafka Transactions (v0.11+) — no duplicate processing
Dead Letter Queue (DLQ)

When a message fails to process (e.g. throws an exception) after N retries, it gets moved to a Dead Letter Queue instead of being lost or endlessly retried. This lets you inspect failed messages, fix the bug, and reprocess them. Essential for production reliability. Both Kafka and RabbitMQ support this pattern.

When to Use Message Queues in System Design

Use queues when: (1) Decoupling — sender shouldn't wait for receiver, (2) Traffic spikes — absorb bursts without dropping requests, (3) Parallel processing — one event triggers multiple independent workflows, (4) Retry logic — failed tasks should be retried automatically, (5) Audit trail — you need a log of all events (use Kafka).

Topic 05
Real-Time Communication

Standard HTTP is request-driven — the client always initiates. For real-time systems (chat, live scores, notifications), you need the server to push data to clients without the client asking. There are four approaches, each with different trade-offs.

Short Polling

Client asks "anything new?" every N seconds. Simple but wasteful — most requests return nothing. High server load.

Use: Simple dashboards, non-critical updates
Long Polling

Client asks, server holds connection open until data is available. When data arrives, server responds and client immediately reconnects. Simulates push.

Use: Chat apps (older), notification systems
WebSockets

Full-duplex persistent connection. Both client and server can send messages anytime. Single TCP connection maintained. Low overhead after setup.

Use: WhatsApp, live chat, multiplayer games, trading
Server-Sent Events (SSE)

One-way persistent stream: server → client only. Client cannot send data back (uses regular HTTP for that). Simpler than WebSockets for push-only needs.

Use: Live news feeds, GitHub activity, notifications
WebRTC

Peer-to-peer media streaming directly between browsers. No server in the media path. Supports video, audio, and arbitrary data. Uses STUN/TURN servers for NAT traversal.

Use: Zoom, Google Meet, video calls, screen share
WebSockets — How They Work
WebSocket Lifecycle
1. HTTP Upgrade Handshake:
   Client → GET /chat HTTP/1.1
            Upgrade: websocket
            Connection: Upgrade
   Server → 101 Switching Protocols  ← connection "upgraded"

2. Persistent Connection:
   Client ──────── "Hello" ──────────► Server  (client sends anytime)
   Client ◄──── "Message from Bob" ─── Server  (server pushes anytime)
   Client ──────── "Read receipt" ─────► Server
   Single TCP connection stays open for entire session

3. Connection closes:
   Either side sends Close frame → connection terminated

WhatsApp uses a custom XMPP protocol over WebSockets/TCP
Each client maintains one persistent connection to an edge server
WebRTC vs WebSockets for Video

WebSockets: great for signalling (who's calling who, call status) but NOT for media (video/audio bytes). Routing all video through a server is expensive and adds latency.

WebRTC: streams video/audio directly peer-to-peer after initial connection. Server is only used for signalling (exchanging IP addresses) and as a TURN relay when P2P isn't possible (firewall/NAT). This is why Zoom can handle millions of calls without routing all video through their servers.

Topic 06
Push vs Pull Architecture

This is one of the most important trade-offs in system design — particularly for feed-based systems like Twitter, Instagram, and Facebook. The question is: when a user posts something, when do their followers see it?

Push (Fan-out on Write)

When Alice posts a tweet, immediately push it to the feeds of all her followers. Their feed is pre-computed and ready.

Read is instant — just fetch pre-built feed from cache.

Write is expensive — if Alice has 50M followers, that's 50M writes immediately.

Used by: Instagram (for most users), TikTok
Pull (Fan-out on Read)

When Bob opens his feed, query all accounts he follows and merge their recent posts on the fly.

Write is cheap — just store the post once in Alice's table.

Read is expensive — if Bob follows 1000 people, that's 1000 queries merged.

Used by: YouTube (subscriptions), some email clients
The Celebrity Problem — Why Twitter Uses Hybrid
Twitter's Hybrid Approach
Regular users (fewer than ~100k followers):
  → Use PUSH (fan-out on write)
  → Tweet stored in all followers' feed caches immediately
  → Fast reads for everyone

Celebrity accounts (millions of followers — Elon Musk, Cristiano Ronaldo):
  → Use PULL (fan-out on read)
  → Tweet stored once. NOT pushed to 100M feeds.
  → When a follower opens the app, celebrity tweets are fetched on-demand
  → Merged with pre-built feed from regular accounts they follow

Why?  Pushing a single tweet to 100 million feeds simultaneously
       would require millions of writes in seconds → system would collapse.
       This threshold is ~100k–500k followers depending on the platform.
Interview Answer Formula

When asked to design a social feed, say: "I'll use a hybrid approach — fan-out on write for regular users with the pre-built feed stored in Redis, and fan-out on read for celebrity accounts above a follower threshold (~100k). This balances write amplification against read latency, since celebrity posts are rare but their fan-out cost is enormous."

Key Takeaway — Phase 03

Every HLD component exists to solve one problem:
Cache → speed.   CDN → proximity.   Load Balancer → distribution.   Queue → decoupling.
Know which problem you're solving before reaching for a tool.
