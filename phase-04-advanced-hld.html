Topic 01
Monolith vs Microservices

Every application starts as a monolith. The question isn't which is better — it's when does the trade-off tip towards microservices. Getting this wrong in either direction is expensive.

Monolith
  • Single deployable unit — one codebase, one process
  • Simple to develop, test, and deploy initially
  • No network latency between components
  • Easy to debug — one log stream, one process
  • Transactions across features are trivial (same DB)
  • Good for: early-stage startups, small teams
  • Becomes: slow to deploy, hard to scale independently
Microservices
  • Each service owns one business domain, deployed independently
  • Teams can work, deploy, and scale independently
  • Each service can use the best language/DB for its job
  • Failure in one service doesn't bring down the entire system
  • Scales horizontally per-service based on load
  • Good for: large teams, high scale, independent domains
  • Becomes: complex distributed systems, harder to debug
Why You Start With Monolith — Always
Evolution of an Architecture
Stage 1 — Startup (0 → 100K users)
  Single monolith on one server
  One PostgreSQL database
  ✓ Ship fast, iterate fast, simple debugging

Stage 2 — Growth (100K → 1M users)
  Monolith scaled horizontally (multiple instances + load balancer)
  Read replicas added to database
  Redis cache layer introduced
  ✓ Still manageable, now handling scale

Stage 3 — Scale (1M+ users, 20+ engineers)
  Problem: 50 engineers pushing to same codebase → deployment conflicts
  Problem: User service and Video service have very different scaling needs
  Problem: A bug in Payments takes down the whole app
  → Time to extract into microservices

Stage 4 — Microservices
  User Service  │  Video Service  │  Payment Service  │  Search Service
  Each owns its DB, scales independently, deploys independently
Strangler Fig Pattern — Migrating Safely

You never rewrite a monolith from scratch — that's called "the second system effect" and it almost always fails. Instead, you strangle it gradually:

1
Identify a Bounded Context

Pick a feature that is mostly self-contained and has clear boundaries — e.g. User Authentication. It should have minimal dependencies on other parts of the monolith.

2
Build the Microservice in Parallel

Write the new Auth Service alongside the monolith. Don't delete monolith code yet. Both exist simultaneously.

3
Route Traffic via a Facade/Proxy

Place a reverse proxy in front. Route /auth/* requests to the new service, everything else to the monolith. Gradually increase traffic percentage to the new service.

4
Delete the Old Code

Once the new service handles 100% of traffic stably, delete the auth code from the monolith. Repeat for the next feature.

Microservices Hidden Costs

Every microservice call that would have been a function call in a monolith is now a network request — adding latency, potential failure, and the need for retries. A transaction that used to be one DB call now requires distributed transactions (two-phase commit or SAGA pattern). Distributed systems are fundamentally harder to debug. Only migrate when the pain of staying monolith exceeds the pain of distributed systems.

Topic 02
Failure Patterns & Resilience

In distributed systems, failures are not exceptional — they are guaranteed to happen. Services crash, networks partition, disks fill up. Resilient systems are designed around the assumption of failure, not the hope of uptime.

Critical Failure Patterns to Know
Single Point of Failure (SPOF)

Any component that, if it fails, takes down the entire system. Examples: a single load balancer, a single database with no replica. Solution: redundancy at every layer.

Cascading Failure

Service A fails → Service B times out waiting for A → B's thread pool exhausts → B fails → Service C fails. One failure propagates across the entire system like dominoes.

Split Brain

Network partition causes two nodes to think they're both the leader. Both accept writes independently. When partition heals, data conflicts exist. Solved by quorum-based leader election.

Thundering Herd

System restarts after crash. Millions of clients reconnect simultaneously and overwhelm the recovering service. Solution: exponential backoff with jitter in clients.

Circuit Breaker Pattern — Preventing Cascades

The circuit breaker is one of the most important resilience patterns. It stops a failing service from being hammered with requests and gives it time to recover.

Circuit Breaker — 3 States
CLOSED  (normal operation)
  │  All requests pass through to Service B
  │  Track failure rate over a rolling window
  │  If failures > threshold (e.g. 50%) → trip to OPEN
  ▼
OPEN    (circuit tripped)
  │  ALL requests immediately rejected (fail fast)
  │  No requests reach Service B — it gets breathing room
  │  Return cached response or error to caller
  │  After timeout (e.g. 30s) → move to HALF-OPEN
  ▼
HALF-OPEN  (testing recovery)
  │  Allow a few test requests through to Service B
  │  If they succeed → reset to CLOSED  (recovered!)
  └  If they fail → back to OPEN         (still down)

Libraries: Netflix Hystrix, Resilience4j, AWS App Mesh
Without Circuit Breaker — The Cascade

Service A calls Service B which is down. Without a circuit breaker, A waits for B's timeout (say 30 seconds) per request. A's thread pool fills with stuck requests. A becomes unresponsive. Service C calls A — same thing. The entire call chain is now frozen, not because they're failing, but because they're all waiting on a dead service.

Other Resilience Patterns
Retry with Exponential Backoff

On failure, retry after 1s, then 2s, then 4s, then 8s. Add random jitter to prevent synchronized retries. Prevents thundering herd on recovery.

Bulkhead Pattern

Isolate resources for different callers. Payment service gets its own thread pool, Search gets its own. If Search overwhelms its pool, Payment is unaffected. Like watertight compartments in a ship.

Timeout

Every service call must have an explicit timeout. Never wait indefinitely. If Service B hasn't responded in 500ms, fail fast and return an error or fallback. Prevents resource exhaustion.

Fallback / Graceful Degradation

When a service fails, return a degraded but useful response. Netflix: if recommendation service is down, show "Top 10 popular movies" instead of personalised picks.

Idempotency

Design operations so repeating them has the same effect as doing them once. Critical for retries — if a payment request is retried, you must not charge the user twice. Use idempotency keys.

Health Checks

Load balancers periodically ping each server's /health endpoint. If a server fails the check, it's removed from rotation automatically. Prevents routing traffic to dead servers.

Topic 03
Docker & Kubernetes

Containerisation is the foundation of modern microservice deployment. Docker packages your app and all its dependencies into a portable container. Kubernetes orchestrates thousands of those containers across a cluster of servers.

Docker — What and Why
The Problem Docker Solves
Without Docker:
  "It works on my machine!" ← the classic problem
  Dev: Node 18, Ubuntu 22 → works fine
  Prod: Node 16, CentOS 7 → crashes
  New developer onboarding: 2 days of environment setup

With Docker:
  App + Node 18 + dependencies + config = one Docker Image
  Same image runs identically on: Dev laptop, CI server, Production
  New developer: `docker-compose up` → running in 5 minutes

Key Concepts:
  Image   = read-only blueprint (like a class)
  Container = running instance of an image (like an object)
  Dockerfile = instructions to build an image
  Registry = storage for images (Docker Hub, AWS ECR)
Dockerfile — Anatomy
FROM    node:18-alpine          # base image
WORKDIR /app                    # working directory inside container
COPY    package*.json ./        # copy dependency files
RUN     npm ci                  # install dependencies (cached layer)
COPY    . .                     # copy source code
EXPOSE  3000                    # document which port the app uses
CMD     ["node", "server.js"]   # command to run when container starts

Each instruction = a layer. Unchanged layers are cached.
This is why COPY package.json comes BEFORE COPY . . (source changes often,
dependencies don't — cache the expensive npm install step)
Kubernetes — Orchestrating at Scale

Once you have hundreds of containers, someone needs to manage them — starting them, restarting crashed ones, scaling them up/down, distributing traffic. That's Kubernetes (K8s).

Kubernetes Architecture
Control Plane (the brain)
  ├─ API Server     — central command, all kubectl commands go here
  ├─ Scheduler      — decides which Node to run a Pod on
  ├─ Controller     — ensures desired state matches actual state
  └─ etcd           — distributed key-value store for cluster state

Worker Nodes (the muscle)
  ├─ Pod            — smallest unit: one or more containers sharing network
  ├─ kubelet        — agent on each node, runs and monitors pods
  └─ kube-proxy     — handles networking rules between pods

Key Resources:
  Deployment      — manages replica sets, rolling updates, rollbacks
  Service         — stable IP/DNS for a set of pods (pods come and go)
  HPA             — Horizontal Pod Autoscaler (scale based on CPU/memory)
  ConfigMap       — inject config without rebuilding image
  Secret          — inject sensitive values (passwords, tokens)
Self-Healing in Action
You declare: "I want 3 replicas of user-service running at all times"

  K8s creates 3 pods → all running  ✓

  Node 2 crashes  →  Pod 2 disappears
  Controller detects: actual=2, desired=3
  Scheduler picks a healthy node
  New Pod 3 created on Node 1      ← automatic, ~10 seconds
  Traffic re-routed away from dead pod immediately

  You don't page anyone. It just heals itself.
Service Discovery in Microservices

In a dynamic environment where pods start and stop constantly, how does Service A find Service B's IP address? K8s Services provide a stable DNS name (e.g. payment-service.default.svc.cluster.local) that always points to live pods, regardless of which physical IPs they have. This is internal service discovery — no hardcoded IPs needed.

Topic 04
Security Patterns

Security in system design is about answering two questions: Who are you? (Authentication) and What are you allowed to do? (Authorization). Getting these right prevents data breaches, unauthorised access, and fraud.

Authentication — Proving Identity
JWT — JSON Web Token Anatomy
A JWT is three base64-encoded parts joined by dots:

  Header.Payload.Signature

Header:  { "alg": "HS256", "typ": "JWT" }
         Which signing algorithm to use

Payload: { "user_id": 42, "role": "admin", "exp": 1700000000 }
         Claims — who this is and when it expires
         NOT encrypted — anyone can read it. Don't put secrets here!

Signature: HMAC_SHA256(base64(header) + "." + base64(payload), SECRET_KEY)
           Server signs it. Only server with the SECRET_KEY can verify it.
           Prevents tampering — if payload changes, signature breaks.

Flow:
  User logs in → Server creates JWT → Returns to client
  Client stores JWT → Sends in every request: Authorization: Bearer <token>
  Server verifies signature → If valid, trusts the claims → No DB lookup needed
  Stateless auth — scales perfectly with horizontal scaling
OAuth 2.0 — Delegated Authorization Flow
"Login with Google" uses OAuth 2.0:

  1. User clicks "Login with Google" on your app
  2. Your app redirects to Google's auth server
  3. User logs in to Google, approves permissions
  4. Google redirects back with an Authorization Code
  5. Your backend exchanges code for Access Token + Refresh Token
  6. Access token used to call Google APIs on user's behalf

Access Token:  short-lived (15 min) — used for API calls
Refresh Token: long-lived (30 days) — used to get new access tokens silently

Key insight: Your app NEVER sees the user's Google password.
             Google handles auth. You just get a token saying "this user approved access".
Authorization — Controlling Access
RBAC — Role Based

Users assigned to roles. Roles have permissions. User → Role → Permission. Simple and widely used. Example: Admin, Editor, Viewer roles in a CMS.

ABAC — Attribute Based

Permissions based on attributes of user, resource, and environment. "Users in Finance dept can read reports marked as Financial during business hours." More flexible than RBAC.

ACL — Access Control List

Per-resource list of who can do what. Like file permissions: Alice has read+write, Bob has read-only, Charlie has no access. Used in Google Drive sharing.

Encryption at Rest

Data is encrypted when stored on disk. Even if someone steals the hard drive, they can't read the data. AWS S3, RDS offer this transparently with AES-256.

Encryption in Transit

Data encrypted while moving between services using TLS. Prevents man-in-the-middle attacks. All internal microservice communication should use mTLS (mutual TLS — both sides verify).

Secrets Management

Never hardcode passwords or API keys in code or Docker images. Use HashiCorp Vault, AWS Secrets Manager, or Kubernetes Secrets. Rotate credentials regularly.

Zero Trust Security Model

Traditional: trust everything inside the corporate network. Zero Trust: trust nothing, verify everything — even internal service-to-service calls are authenticated and authorised. Every request, regardless of source, must prove its identity. This is the modern standard, especially for microservices architectures where services are constantly talking to each other.

Topic 05
Monitoring & Observability

In distributed systems, when things go wrong (and they will), you need to know what happened, where, and why. Observability is built on three pillars: Metrics, Logs, and Traces. Each answers a different question.

The Three Pillars of Observability
METRICS   — What is the system doing RIGHT NOW?
  Numbers over time: CPU 78%, 500 req/sec, p99 latency=240ms, error rate=0.1%
  Fast to query. Great for dashboards and alerting thresholds.
  Tools: Prometheus (collect) + Grafana (visualise)

LOGS      — What HAPPENED and WHEN?
  Timestamped text records of discrete events
  "User 42 logged in", "Payment failed: insufficient funds", "DB query took 2.3s"
  Tools: ELK Stack (Elasticsearch + Logstash + Kibana), AWS CloudWatch

TRACES    — HOW did a request flow through all services?
  A single user request may touch 12 microservices in 200ms
  Distributed tracing shows the full journey with timing at each hop
  Reveals: which service is the bottleneck, where latency is added
  Tools: Jaeger, Zipkin, AWS X-Ray
Monitoring Stack — Tool by Tool
Metrics Collection
Prometheus

Scrapes metrics from services on a schedule. Stores as time-series. Has a powerful query language (PromQL). The industry standard for Kubernetes environments.

Visualisation
Grafana

Dashboards for Prometheus (and other sources). Build real-time charts of any metric. Alerting rules trigger PagerDuty/Slack notifications when thresholds are breached.

Log Aggregation
ELK Stack

Logstash collects and transforms logs. Elasticsearch indexes and stores them. Kibana provides search UI. Search across millions of log lines in milliseconds.

Distributed Tracing
Jaeger / Zipkin

Instruments each service to emit span data. Assembles spans into full traces. See the exact path of a slow request across 10 services with millisecond precision.

Cloud Native
AWS CloudWatch

AWS-native monitoring for EC2, Lambda, RDS, etc. Logs, metrics, traces, alarms in one place. First choice for AWS-centric architectures.

Anomaly Detection
Alerts + SLOs

SLI: what you measure (p99 latency). SLO: target value (<200ms 99.9% of time). SLA: contractual promise to users. Alerts fire when SLOs are at risk of being breached.

SLI / SLO / SLA — Don't Confuse Them

SLI (Service Level Indicator) — the metric you measure. e.g. "% of requests with latency < 200ms"

SLO (Service Level Objective) — the target you set internally. e.g. "SLI must be ≥ 99.9% over 30 days". This is your internal reliability goal.

SLA (Service Level Agreement) — the contractual promise to paying customers. e.g. "We guarantee 99.9% uptime or we issue credits." Your SLO must be stricter than your SLA — you can't keep an SLA you're not internally targeting.

Topic 06
System Design Trade-offs Master Reference

In any system design interview, there is no single right answer. Every decision trades one property for another. This is your master reference — know these cold.

Decision Choose A when... Choose B when...
SQL vs NoSQL
Data is relational, need ACID transactions, schema is stable. (Payments, orders, inventory)
Need massive scale, flexible schema, high write throughput. (Feeds, logs, user activity)
Consistency vs Availability
Data must always be correct — stale is worse than unavailable. (Bank balance, stock count)
System must always respond — slightly stale is better than error. (Social feed, notifications)
Monolith vs Microservices
Early stage, small team (<10 engineers), moving fast, simple domain.
Large team (20+), independent scaling needs, multiple distinct domains, high deployment frequency.
Push vs Pull (Fan-out)
Most users have modest follower counts, read speed is critical. (Pre-build feeds)
High celebrity/influencer ratio, write amplification is too expensive. (Compute feed on read)
Sync vs Async
Caller needs the result immediately to continue. (Search results, login, payment confirmation)
Task can happen in background. (Send email, resize image, update feed, push notification)
Strong vs Eventual Consistency
Every read must see the latest write. Accept higher latency for correctness.
Can tolerate brief staleness for lower latency and higher availability at scale.
Serverless vs Serverful
Spiky, infrequent workloads. Want zero ops overhead. Short-running functions.
Sustained high traffic. Long-running processes. Need fine-grained control or lower per-request cost.
Cache Write-Through vs Write-Back
Data consistency is critical. Can absorb slower write latency.
Write speed is paramount. Can tolerate small window of potential data loss.
Normalisation vs Denormalisation
Write-heavy system. Storage efficiency matters. Data integrity critical. (Transactions)
Read-heavy system. Need fast queries without joins. (Feed, product catalog, analytics)
Vertical vs Horizontal Scaling
Stateful app (hard to distribute), simple architecture, database servers (easier to scale up).
Stateless app, need fault tolerance, high throughput at commodity cost. Web/API servers.
The Interview Formula for Every Trade-off

Structure every trade-off answer as: "I'll choose [X] over [Y] because our system's primary constraint is [requirement]. The trade-off is [what we sacrifice], which is acceptable because [why it doesn't matter for our use case]."

This shows the interviewer you understand there's no free lunch — you're making a deliberate, justified choice, not guessing.

Key Takeaway — Phase 04

Advanced HLD is not about more components.
It's about knowing when things fail,
how to contain the blast radius,
and how to justify every choice under cross-examination.
