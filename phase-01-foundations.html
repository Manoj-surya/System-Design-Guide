Topic 01
How the Internet Works
DNS Resolution

DNS (Domain Name System) is the internet's phonebook. When you type google.com, your computer has no idea what IP address that maps to. DNS resolves the domain name into an IP address step by step.

DNS Resolution Flow
Step 1.  Browser cache → does it already know this domain?
Step 2.  OS cache / /etc/hosts file
Step 3.  Query sent to Recursive Resolver (usually your ISP)
Step 4.  Resolver asks Root DNS Server  →  "Who handles .com?"
Step 5.  Root says   →  "Ask the .com TLD nameserver"
Step 6.  TLD says    →  "Ask google.com's own nameserver"
Step 7.  google.com nameserver returns IP: 142.250.80.46
Step 8.  Browser connects to that IP directly
System Design Relevance

DNS can act as a basic load balancer. By returning different IP addresses for the same domain (DNS Round Robin), you can distribute traffic across multiple servers without a dedicated load balancer. This is how large sites handle initial request distribution across data centers globally.

TTL — Time To Live

Each DNS record has a TTL (e.g. 300 seconds). Results are cached at every step for that duration. This is why DNS changes can take time to propagate globally — old records are cached everywhere until TTL expires.

Request-Response Cycle

Every interaction on the web is a client-initiated request followed by a server response. The server never pushes data unless you use WebSockets or SSE (covered in Phase 3).

HTTP Request / Response Structure
REQUEST  (Client → Server)
  ├─ Method:   GET / POST / PUT / DELETE
  ├─ URL:      /api/users/123
  ├─ Headers:  Content-Type, Authorization, ...
  └─ Body:     { "name": "Alice" }   (for POST/PUT)

RESPONSE (Server → Client)
  ├─ Status:   200 OK / 404 Not Found / 500 Error
  ├─ Headers:  Content-Type, Cache-Control, ...
  └─ Body:     { "id": 123, "name": "Alice" }
TCP vs UDP

These are the two core transport layer protocols — the rules governing how packets of data travel between machines across a network.

Property TCP UDP
Full NameTransmission Control ProtocolUser Datagram Protocol
ConnectionConnection-oriented (handshake required)Connectionless (fire and forget)
ReliabilityGuaranteed delivery, ordered packetsNo guarantee, packets can drop
SpeedSlower — overhead of ACKs & retransmitsFaster — minimal overhead
Error CheckingYes — retransmits lost packetsMinimal checksum only
Use CasesHTTP, emails, file transfer, bankingVideo calls, live streaming, gaming, DNS
TCP 3-Way Handshake
Client  ──── SYN ────────────►  Server   "I want to connect"
Client  ◄─── SYN-ACK ────────  Server   "OK, I acknowledge you"
Client  ──── ACK ────────────►  Server   "Great, we're connected!"
                                           ← Data flows now →
Why UDP for Video Calls?

If one video frame drops during a call, you don't want TCP to pause everything, retransmit that frame, and resume — the moment has passed. A brief visual glitch is far better than a frozen call. With UDP, latency matters more than perfection. This is why WhatsApp calls, Zoom, and gaming all use UDP.

HTTP Versions — 1.1 vs 2 vs 3
VersionKey FeatureProblem it Solved
HTTP/1.1 Persistent connections (keep-alive) Previously every request opened & closed a new TCP connection
HTTP/2 Multiplexing — multiple requests over one TCP connection simultaneously HTTP/1.1 head-of-line blocking: requests queued, each waits for previous
HTTP/3 Runs over QUIC (UDP-based protocol) instead of TCP TCP's transport-layer head-of-line blocking; faster on lossy/mobile networks
HTTP/1.1 vs HTTP/2 — Visual Difference
HTTP/1.1  — requests queue up, each waits:
  Req 1  ──────────────────►  Res 1
                                       Req 2  ────►  Res 2  ← had to wait

HTTP/2   — all requests fly simultaneously on ONE connection:
  Req 1  ──►
  Req 2  ──►  same TCP connection
  Req 3  ──►
  
HTTP/3   — uses QUIC over UDP, no TCP head-of-line blocking at all:
  Even if one packet drops, other streams unaffected
HTTPS & TLS Handshake

HTTPS = HTTP + TLS (Transport Layer Security). TLS encrypts all communication so nobody in the middle (man-in-the-middle attack) can read or tamper with the data.

TLS Handshake Flow
1. Client  →  "Hello! I support these encryption methods: AES-256, ..."
2. Server  →  "Use AES-256. Here's my SSL Certificate (public key inside)"
3. Client  →  Verifies certificate with a Certificate Authority (CA)
4. Client  →  Generates a session key, encrypts with server's public key, sends it
5. Server  →  Decrypts with its private key — both now share the session key
6.          All further data is encrypted with this shared session key
Asymmetric Encryption

Used only during handshake. Public key encrypts, private key decrypts. Secure but computationally slow. Examples: RSA, ECC.

Symmetric Encryption

Used for actual data transfer. Both sides share the same key. Much faster. Examples: AES-256, ChaCha20.

Topic 02
Threads, Processes & Concurrency
Processes vs Threads
Mental Model
Process  =  A running program with its OWN memory space
           e.g. Chrome.exe, node.exe, python.exe

Thread   =  A unit of execution INSIDE a process
           Shares memory with other threads in the same process
           Lighter and cheaper to create than a process
Restaurant Analogy

Process = the entire restaurant (its own building, kitchen, address, resources). Thread = individual waiters working inside that restaurant, all sharing the same kitchen. One restaurant can have many waiters. Waiters are fast to hire/fire compared to opening a new restaurant.

The Shared Memory Problem

Because threads share memory, if two threads try to read and write the same variable at the same time, you get a race condition — unpredictable, buggy behaviour. This is why thread safety, locks, and synchronisation matter (covered in Phase 5 LLD).

Context Switching

A CPU core can only execute one thread at a time. To simulate running many threads simultaneously, the OS rapidly switches between them — saving and restoring each thread's state. This is called a context switch.

Context Switching Flow
Thread A runs  ──── 10ms ────►  OS saves A's state (registers, stack pointer)
Thread B runs  ──── 10ms ────►  OS saves B's state
Thread A runs  ──── 10ms ────►  OS restores A's saved state, resumes

Every save/restore = overhead.
Too many threads → too much context switching → performance collapses.
This is why Node.js uses ONE thread with an event loop instead of one thread per request.
Synchronous vs Asynchronous
Synchronous (Blocking)
Thread is BLOCKED while waiting for I/O — doing nothing useful:

Task 1: Read file from disk  ────────────────────► done
Task 2:                      ← waiting... waiting...     ► starts now

Thread sits idle for hundreds of milliseconds waiting for disk.
Terrible for servers handling thousands of requests.
Asynchronous (Non-Blocking)
Thread registers a callback and moves on immediately:

Task 1: Read file   ──►  (delegate to OS, register callback)  ──►  callback fires when done
Task 2:             ──────────────────────────────────────────►  done (started immediately!)

Thread stays free to handle other work. This is how Node.js serves
10,000 concurrent connections with a single thread.
The Event Loop (Node.js model)

Node.js uses a single-threaded event loop for async I/O. Understanding this helps you understand why blocking the event loop with heavy computation is catastrophic.

Event Loop Architecture
         ┌──────────────────────────────┐
         │         Call Stack           │  ← executes sync code top-to-bottom
         └──────────────┬───────────────┘
                        │ empty?
         ┌──────────────▼───────────────┐
         │          Event Loop          │  ← watches: is call stack empty?
         └──────────────┬───────────────┘
                        │ yes → pick next callback
         ┌──────────────▼───────────────┐
         │        Callback Queue        │  ← finished async ops wait here
         └──────────────────────────────┘

Example:
  setTimeout(() => console.log("B"), 0)    // goes to callback queue
  console.log("A")                           // runs immediately on stack
  // Output: A  then  B                    // even with 0ms delay
Critical Gotcha

If you run a heavy CPU computation synchronously in Node.js (like sorting a million items), it blocks the entire event loop. No other requests can be handled until it finishes. This is why CPU-heavy work should be offloaded to worker threads or a separate service.

Topic 03
Serverless vs Serverful
Serverful — AWS EC2

You rent a virtual machine. You control the OS, runtime, dependencies, and scaling rules. Pay 24/7 whether traffic exists or not. Full control, full responsibility.

Serverless — AWS Lambda

Upload just your function. AWS handles the server, OS, scaling, and patching. Pay only per invocation — scales to zero. Cold start latency on first call.

Trade-off Comparison
                    SERVERFUL (EC2)         SERVERLESS (Lambda)
Control         ──  Full (OS, runtime)       None (managed by AWS)
Cost model      ──  Pay 24/7                 Pay per invocation
Scaling         ──  Manual / auto-scaling    Automatic (instant)
Cold starts     ──  None                     Yes (first call slow)
Long processes  ──  ✓ Unlimited              ✗ Max 15 min timeout
Vendor lock-in  ──  Low                      High
Ideal for       ──  Sustained high traffic   Spiky / infrequent tasks
When to Use Each — System Design Context

Serverful: Netflix's main video streaming servers — constant high traffic, long-running processes, cost-effective at scale.

Serverless: Resize a thumbnail when a user uploads a photo on Instagram, send a welcome email after signup, process a webhook. Infrequent, short, event-triggered tasks.

The Cold Start Problem

When a Lambda function hasn't been called recently, AWS needs to spin up a new container for it. This first invocation can take 100ms–2s extra. For latency-sensitive applications (like real-time APIs), this is a significant trade-off. Solutions: provisioned concurrency (keep containers warm, but costs more), or just use serverful for latency-critical paths.

Topic 04
Vertical vs Horizontal Scaling
Vertical Scaling (Scale Up)

Add more power to the same single machine — more CPU cores, more RAM, faster storage.

Vertical Scaling
Start:   Server  [4 CPU · 8GB RAM · 100GB SSD]
           ↓ upgrade the same machine
After:   Server  [32 CPU · 256GB RAM · 2TB SSD]

Pros:  Simple. No code changes. No distributed complexity.
Cons:  Hard limit exists. Single point of failure.
       Requires downtime to upgrade. Very expensive at top end.
Horizontal Scaling (Scale Out)

Add more machines and distribute load between them using a load balancer.

Horizontal Scaling
                   ┌──────────────────┐
                   │   Load Balancer  │
                   └──────┬───────────┘
              ┌───────────┼───────────┐
              ▼           ▼           ▼
          Server 1    Server 2    Server 3
         [4C·8GB]    [4C·8GB]    [4C·8GB]

Pros:  No theoretical limit. Fault tolerant.
       Cheaper commodity hardware. Zero-downtime scaling.
Cons:  Requires stateless app design.
       Distributed systems complexity. Needs a load balancer.
The Stateless Requirement — Critical Concept

For horizontal scaling to work, any server must be able to handle any request. This means a server cannot store session data in its own memory — because the next request from the same user might hit a different server.

Solution: Store shared state in an external store like Redis that all servers can access. This is why almost every horizontally-scaled system uses Redis for sessions and caching.

FactorVertical ScalingHorizontal Scaling
ComplexitySimpleComplex (distributed systems)
LimitHardware ceilingTheoretically unlimited
Fault toleranceNone (SPOF)High (redundancy built-in)
Cost at scaleVery expensiveCost-effective (commodity)
Downtime to scaleYes (restart required)No (add servers live)
Used bySmall apps, databasesNetflix, Twitter, WhatsApp
Real World: WhatsApp Architecture

WhatsApp handled 900 million users with only ~50 engineers — largely because they used horizontal scaling with Erlang, which can run millions of lightweight processes on one machine, and then scaled out horizontally across thousands of servers. Each server is stateless; presence and session data lives in a shared registry.
